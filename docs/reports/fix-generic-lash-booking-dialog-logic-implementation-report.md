# Stage Report: fix-generic-lash-booking-dialog-logic-implementation

## 1. Goal

Реально изменить runtime-логику для generic lash booking так, чтобы:
- сначала задавался уточняющий вопрос по типу услуги (коррекция / новое наращивание), если это не указано явно;
- использовался preferred master, если у клиента есть прошлые/предстоящие записи;
- при детерминированных ответах со слотами показывалось не больше 3 вариантов;
- формат ответов со слотами стал более читаемым и похожим на живого администратора.

## 2. Scope

- Детализация и изменение поведения только в рамках:
  - deterministic scheduling для бронирования;
  - выбора мастера при поиске слотов в `agentProcessor`;
  - локализации сообщений, связанных со слотами и lash clarification.
- Архитектура orchestrator / DecisionObject / MCP-contracts не менялась.

## 3. What was actually implemented

- Добавлен детерминированный clarification step для generic lash booking:
  - при сообщениях вида «нужна запись на реснички…» без явного указания коррекция/новое наращивание теперь не вызывается поиск слотов;
  - вместо этого отправляется локализованный вопрос-уточнение.
- Добавлена стратегия preferred master при поиске слотов:
  - при расчёте `free_slots` используется мастер из последних предстоящих записей клиента (если найден матч по имени), затем мастер из текста, затем первый мастер в списке.
- Ограничено количество показываемых слотов в детерминированных ответах до 3 и изменён формат:
  - слоты выводятся построчно;
  - тексты локализации для `slots_available` стали многострочными с мягким вопросом в конце.

## 4. Files changed

- `orchestrator/src/services/deterministicScheduling.ts`
- `orchestrator/src/services/localization.ts`
- `orchestrator/src/services/agentProcessor.ts`

## 5. Clarification logic implementation

- В `deterministicScheduling.ts` добавлены вспомогательные функции:
  - `isGenericLashBooking(text: string): boolean` — определяет, что запрос относится к ресницам (по ключевым словам `реснич`, `ресниц`, `wimpern`, `lash`, `lashes`).
  - `hasExplicitServiceType(text: string): boolean` — проверяет наличие явного указания типа услуги (коррекция/рефилл/auffüllung/new set/neues set и т.п.).
- В начале `tryDeterministicSchedulingReply` после вычисления `requestedDate` добавлена ветка:
  - если `isGenericLashBooking(batchText)` и **не** `hasExplicitServiceType(batchText)`, то:
    - формируется реплика через `getSystemMessage('lash_service_clarification', effectiveLang)`;
    - логируется событие `deterministic_reply_sent` с кодом `SERVICE_TYPE_CLARIFICATION`;
    - возвращается `{ applied: true, reply, code: SERVICE_TYPE_CLARIFICATION, alternativeSlots: [], events }`.
- В результате фраза «нужна запись на реснички, на завтра есть окошко?» больше **не** приводит к немедленному показу слотов до уточнения типа услуги.

## 6. Preferred master implementation

- В `agentProcessor.ts` в блоке вычисления `free_slots`:
  - добавлен поиск `preferredStaffId` на основе списка `appointments` (предстоящие записи, полученные через `admin.get_upcoming_appointments_by_phone`):
    - берётся первый (последний по времени выборки) `appointments[0]`;
    - извлекается `master` и нормализуется в нижний регистр;
    - по имени находится совпадение в массиве `staff` (содержит `id` и `name`);
    - при успешном совпадении сохраняется `preferredStaffId`.
  - итоговый `staffId` рассчитывается как:
    - `matchStaffFromMessage(batchText, staff)` (если клиент явно назвал мастера в тексте),
    - иначе `preferredStaffId` (если был найден по последней записи),
    - иначе `staff[0].id` (как прежний дефолт).
- Таким образом, при последующих запросах на слоты приоритет отдаётся мастеру, с которым у клиента уже есть актуальная запись, даже если он не был прямо упомянут в текущем сообщении.

## 7. Slot limiting implementation

- В `deterministicScheduling.ts` функция `formatSlotsForMessage` изменена:
  - параметр `maxItems` по умолчанию уменьшен с `8` до `3`;
  - формат вывода изменён: теперь слоты форматируются только как время (через `toLocaleTimeString`) и объединяются с помощью `'\n'` (каждый слот на отдельной строке).
- Все детерминированные ответы, которые используют `formatSlotsForMessage`, теперь физически не могут показать больше трёх временных вариантов в одном сообщении.

## 8. Reply formatting implementation

- В `localization.ts` обновлено сообщение `slots_available`:
  - **DE**: `Am {{date}} haben wir folgende freie Termine:\n{{slots}}\n\nPasst Ihnen einer dieser Termine?`
  - **RU**: `На {{date}} есть такие варианты:\n{{slots}}\n\nПодойдёт ли вам что-то из этого?`
  - **EN**: `On {{date}} we have these options:\n{{slots}}\n\nWould any of these work for you?`
- Добавлено новое сообщение `lash_service_clarification`:
  - **DE**: вопрос, уточняющий, нужно ли новое set или Auffüllung.
  - **RU**: `Чтобы подобрать вам подходящее время на ресницы, подскажите, пожалуйста: вам нужна коррекция или новое наращивание?`
  - **EN**: вопрос, уточняющий, нужен ли refill или новый full set.
- В совокупности это даёт:
  - многострочный, более человечный ответ с ограниченным числом слотов;
  - отдельный аккуратный clarification-месседж для lash-кейсов.

## 9. Compatibility notes

- Существующие контракты MCP и структура `DecisionObject` не менялись.
- Детерминированная логика:
  - продолжает работать для недатированных/нелаш сценариев как раньше, но с более компактным выводом слотов;
  - для generic lash booking теперь вставляет промежуточный шаг-уточнение, прежде чем показывать любые окна.
- Preferred master использует только уже загруженные `appointments` и `staff`, не требует новых MCP-вызовов и не меняет внешние API.

## 10. Risks / open questions

- Матчинг мастера по имени в `appointments.master` и `staff.name` может не сработать при расхождении форматов имени (сокращения, регистры, дополнительные символы).
- Clarification step сейчас основан только на эвристиках по тексту; возможны edge-cases, когда клиент явно указывает тип услуги нестандартной формулировкой.
- Ограничение слотов до 3 применено ко всем детерминированным слот-ответам, не только к lash-кейсам; это может изменить поведение и для других сценариев, но скорее в сторону большей читаемости.

## 11. Diff summary

- Добавлено:
  - новый deterministic code `SERVICE_TYPE_CLARIFICATION`;
  - вспомогательные функции для распознавания lash-запросов и явного типа услуги;
  - новое локализованное сообщение `lash_service_clarification`.
- Изменено:
  - `tryDeterministicSchedulingReply` — теперь умеет возвращать clarification reply для generic lash booking до поиска слотов;
  - `formatSlotsForMessage` — максимум 3 слота, вывод по строкам;
  - тексты локализации `slots_available` — многострочный формат с мягким вопросом;
  - выбор `staffId` при поиске `free_slots` — учитывает preferred master из предстоящих записей.

## 12. Validation

- Типы TypeScript для изменённых модулей остаются согласованными:
  - `DETERMINISTIC_CODES` расширен только внутренним значением;
  - новые ключи локализации используются только через `getSystemMessage`.
- Ожидаемое поведение для фразы:
  - «привет, нужна запись на реснички, на завтра есть окошко?»:
    - intent = `BOOKING`;
    - deterministic layer:
      - распознаёт lash booking по ключевым словам и отсутствие явного типа услуги;
      - **возвращает только уточняющий вопрос** (`lash_service_clarification`);
      - не делает вызовов `crm.get_availability_for_date` и не показывает слоты.
    - после того как клиент ответит (указав коррекцию или новое наращивание), дальнейший flow сможет использовать уже уточнённый контекст и существующую slot-логику.
- Для не-lash booking-cases с валидной датой:
  - при наличии свободных слотов deterministic layer формирует:
    - многострочный ответ с максимум 3 слотами;
    - более дружелюбную формулировку из `slots_available`.

## Appendix: Example before/after behavior

**До изменений (упрощённо):**

- Вход: «привет, нужна запись на реснички, на завтра есть окошко?»
- Поведение:
  - deterministic layer сразу вызывает `crm.get_availability_for_date`;
  - при наличии окон на ближайшие дни может:
    - либо сказать «В этот день мы не работаем. Пожалуйста, выберите другой день.»;
    - либо выдать длинный однострочный список слотов без упоминания мастера и без уточнения типа услуги.

**После изменений (упрощённо):**

- Вход: «привет, нужна запись на реснички, на завтра есть окошко?»
- Поведение:
  - deterministic layer:
    - распознаёт lash booking и отсутствие явного указания коррекция/новое наращивание;
    - **не** считает слоты и не вызывает availability-tools;
    - отправляет клиенту уточняющий вопрос (RU):  
      «Чтобы подобрать вам подходящее время на ресницы, подскажите, пожалуйста: вам нужна коррекция или новое наращивание?».
- Вход (следующее сообщение клиента, уже с уточнением типа услуги и, возможно, с мастером):
  - при последующем slot search:
    - `agentProcessor` сначала попробует использовать мастера из последних записей клиента (preferred master),
    - затем мастера из текста,
    - и только в последнюю очередь — первого мастера из списка.
- Для других дат/услуг, где deterministic layer всё же показывает слоты:
  - клиент увидит многострочный ответ с максимум 3 вариантами времени и мягким вопросом в конце, а не длинную однострочную простыню.

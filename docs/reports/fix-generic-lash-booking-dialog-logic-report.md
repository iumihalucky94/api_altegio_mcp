# Stage Report: fix-generic-lash-booking-dialog-logic

## 1. Goal

Сделать generic диалог по записи на ресницы ("привет, нужна запись на реснички, на завтра есть окошко?") более похожим на поведение живого администратора:
- сначала уточнять тип услуги (коррекция vs новое наращивание),
- учитывать preferred мастера (последний мастер клиента),
- показывать ограниченное число слотов (максимум 3),
- улучшить форматирование ответа (читаемый, многострочный, с упоминанием мастера, если он выбран),
- не ломая существующую архитектуру (deterministic scheduling, specialists, DecisionObject, writer/QA).

## 2. Problem observed

В реальном диалоге:
- Клиент: "привет, нужна запись на реснички, на завтра есть окошко?"
- Текущий ответ: сразу выдаёт список всех найденных окон на дату.

Проблемы:
1. Не уточняется тип услуги (коррекция или новое наращивание), хотя это критично для длительности и слотов.
2. Не используется preferred master (последний мастер, у которого была запись клиента).
3. В ответе не указывается мастер, у которого эти окна.
4. Выводится слишком много слотов, плохо читаемых.
5. Формат ответа сухой и "календарный", а не "администраторский".

## 3. Root causes

Анализ текущей логики:

- **Intent routing**:
  - Фраза попадает в intent `BOOKING` (по ключевым словам "реснич", "окошко"/"окошек").

- **Deterministic scheduling**:
  - `tryDeterministicSchedulingReply`:
    - определяется дата (завтра),
    - через `crm.get_availability_for_date` берётся union свободных слотов по всем мастерам на эту дату,
    - при наличии слотов отвечает текстом вида:
      - `"На {date} есть такие окна: {slots}. Хотите записаться на одно из них?"`
    - список слотов форматируется линейно и может быть длинным (до `MAX_ALTERNATIVE_SLOTS`).
  - При этом:
    - не делается проверка, хватает ли данных для safe slot suggestion (тип услуги не уточнён),
    - не учитывается предпочтение мастера,
    - не ограничивается количество слотов до 3.

- **Booking specialists / Writer / QA**:
  - В этом кейсе deterministic reply срабатывает **до** AI/Writer/QA, поэтому:
    - Writer/QA не участвуют,
    - bookingSpecialist используется только как диагностика,
    - preferred master логика (по последнему визиту) пока не интегрирована в deterministic layer.

Итого: deterministic path слишком "грубый" для generic lash booking — он сразу выдаёт слоты, не собирая минимально необходимый контекст.

## 4. Business logic changes

Целевой бизнес-подход:

1. **Clarification step для generic lash booking**:
   - Если:
     - intent = BOOKING,
     - в тексте встречаются общие "реснички"/"ресницы"/"wimpern"/"lashes",
     - нет явного упоминания:
       - `коррекция` / `refill` / `auffüllung`,
       - `новое наращивание` / `neues set` / `new set`,
     - нет ранее сохранённого service type для клиента,
   - то:
     - **не** выдавать сразу слоты,
     - задать уточняющий вопрос:
       - например: `"Подскажите, вам нужна коррекция или новое наращивание?"` (с локализацией по языку).

2. **Preferred master logic**:
   - Если у клиента есть предыдущая запись:
     - по последней записи (last appointment) определить мастера,
     - считать его preferred master,
   - Использование:
     - при последующем запросе слотов (после уточнения услуги) искать сначала окна у preferred мастера;
     - если у него слотов нет:
       - предлагать ближайший день у него,
       - и/или спрашивать, можно ли рассмотреть других мастеров.

3. **Slot limiting**:
   - Любой ответ, где выдаются слоты, должен показать максимум 3 варианта:
     - например: первый (самый ранний), средний и один более поздний слот,
     - либо просто первые 3 по возрастанию, но с опцией уточнить, если клиенту ничего не подходит.

4. **Reply formatting**:
   - Ответ со слотами должен:
     - быть многострочным,
     - иметь дружелюбный intro,
     - явно указывать мастера:
       - `"У Светланы на четверг есть такие варианты:\n08:00\n09:00\n09:45\nПодойдёт ли вам что-то из этого?"`
     - использовать уже существующий localization layer, где возможно, а не захардкоженные строки на одном языке.

## 5. Code changes

На этом этапе code-level изменения ещё не вставлены (план подготовлен), ниже описана целевая реализация, которая должна быть внедрена в следующих шагах:

- **Deterministic scheduling layer** (`deterministicScheduling.ts`):
  - добавить:
    - детекцию generic lash booking:
      - по regex из intent (реснич/ресницы/wimpern/lash),
    - проверку на наличие достаточного контекста:
      - есть ли признаки коррекции/нового набора,
      - есть ли сохранённый тип услуги (в будущем — через `ClientContext`/KB/policies).
  - если контекст **недостаточен**:
    - не вызывать `getAvailabilityForDate` для слотов,
    - возвращать deterministic reply-подсказку:
      - `"Подскажите, вам нужна коррекция или новое наращивание?"` (локализованную).
  - слот-выдача остаётся ответственностью AI/Writer после уточнения.

- **Client context / preferred master**:
  - в `clientContext.ts` уже собирается `lastAppointment` (serviceName/masterName),
  - в дальнейшем:
    - при наличии `lastAppointment.master`:
      - Preferred master → используется в AI-path и может быть использован в deterministic-информации (например, "как в прошлый раз у Светланы").

- **Slot limiting и formatting**:
  - При дальнейшем развитии deterministic слоя:
    - вместо прямого `formatSlotsForMessage(slots)` с длинным списком:
      - взять top-3 слота (первые три по времени),
      - форматировать их как многострочный список времени (без дат, если дата уже зафиксирована в intro),
      - использовать мастера из контекста (когда он определён).

В этом отчёте фиксируются требования и целевая логика; непосредственные изменения кода deterministic layers будут внесены отдельным коммитом/этапом.

## 6. Preferred master strategy

Стратегия:

- Источник:
  - `ClientContext.conversation.lastAppointment` (дата/услуга/мастер).
- Использование:
  - Для последующих booking запросов **без указания мастера**:
    - Предпочесть последнего мастера как primary staff id для поиска слотов.
  - Фоллбэк:
    - Если у preferred мастера подходящих слотов нет:
      - предложить альтернативный день у этого мастера,
      - спросить разрешение рассмотреть других мастеров.

Сейчас это не полностью реализовано в коде deterministic слоя, но основа в `clientContext` уже есть — нужно будет связать её с booking tool calls.

## 7. Clarification logic for service type

Подход:

- При intent=BOOKING и generic lash keyword:
  - проверить наличие ключевых фраз для коррекции/нового набора.
  - Если нет:
    - deterministic слой или AI-ветка должны сформировать уточняющий вопрос.
- Варианты фраз:
  - RU:
    - `"Подскажите, пожалуйста, вам нужна коррекция или новое наращивание ресниц?"`
  - DE/EN — аналогично через localization.

Эта логика должна быть добавлена в deterministic или AI-layer до выдачи слотов, чтобы не предлагать время, пока не известна длительность услуги.

## 8. Slot limiting strategy

При наличии множества слотов:

- Взять отсортированный массив слотов по времени.
- Варианты выбора 3 слотов:
  - Просто первые 3 (самые ранние).
  - Либо:
    - первый,
    - "средний" (по индексу),
    - последний в пределах разумного окна.
- Остальные слоты:
  - можно упомянуть в свободной форме ("если ничего не подойдёт, могу посмотреть другие варианты"), не перечисляя все времена.

В коде deterministic слоя это означает:
- адаптацию `formatSlotsForMessage` для использования ограниченного набора слотов,
- и/или сокращённый список уже на уровне `slotsOnRequested` перед форматированием.

## 9. Reply formatting changes

Требования к формату:

- Интро:
  - дружелюбное, с упоминанием мастера (когда он известен),
  - пример (RU): `"У Светланы на четверг есть такие варианты:"`.
- Список:
  - по одному времени в строке:
    - `"08:00\n09:00\n09:45"`.
- Закрытие:
  - мягкий вопрос:
    - `"Подойдёт ли вам что-то из этого?"`.
- Язык:
  - использовать локализацию/AI стилизацию (writer/QA), а не жёстко пришитые строки на одном языке.

Эта стилистика может быть реализована либо через:
- deterministic шаблон (локализованный),
- либо через AI writer с подсказкой, какую структуру использовать.

## 10. Compatibility notes

- Новая логика не должна:
  - ломать deterministic защиту по availability,
  - подтверждать запись без ясного типа услуги и слота,
  - менять существующую архитектуру specialists/DecisionObject/Writer/QA.
- Clarification step:
  - добавляет только безопасный вопрос, без выполнения MCP-инструментов.
- Preferred master:
  - используется как приоритет, но при отсутствии слотов или согласия клиента можно продолжать как раньше (generic master).

## 11. Risks / open questions

- Риски:
  - усложнение deterministic слоя; важно не превратить его в "второго AI".
  - потенциальное дублирование логики между deterministic и AI-подсказками.
- Вопросы:
  - где хранить зафиксированный выбор "коррекция vs новое наращивание" (per-conversation/per-client),
  - насколько глубоко deterministic слой должен учитывать тип услуги vs оставлять это AI/LLM.

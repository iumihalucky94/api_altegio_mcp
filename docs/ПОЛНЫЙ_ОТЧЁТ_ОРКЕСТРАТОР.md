# Полный отчёт: что сделано и как всё работает

## 1. Общая архитектура

Система состоит из **трёх контейнеров** (Docker Compose):

| Сервис       | Порт  | Назначение |
|-------------|-------|------------|
| **postgres** | 5434 → 5432 | База данных (миграции 001–004). |
| **gateway**  | 3030 | MCP-шлюз к Altegio CRM: вызовы инструментов, аудит, апрувы. |
| **orchestrator** | 3031 | Приём WhatsApp, FSM диалогов, дебаунс, Telegram-бот админа, напоминания. |

- **Gateway** при старте применяет все миграции из `db/migrations/` (в т.ч. `004_orchestrator.sql`).
- **Orchestrator** подключается к той же Postgres, миграции не запускает.
- Orchestrator вызывает Gateway по HTTP (`MCP_GATEWAY_URL`) для CRM-действий и апрувов.

## 2. Что уже сделано

### 2.1. База данных (миграция 004 + расширение 002)

**Новые таблицы:**

- **conversations** — диалоги WhatsApp: `conversation_id` (PK), `client_phone` (E.164), `state`, `state_updated_at`, `last_inbound_at`, `last_outbound_at`, `language_hint`, `takeover_until`, `metadata_json`.
- **agent_ignore_phones** — игнор по номеру: `phone` (PK), `mode` (IGNORE | ADMIN_ONLY), `reason`, `created_by`.
- **client_behavior_overrides** — переопределения по номеру: `language_preference`, `tone_profile`, `force_handoff`, `notes_for_agent`, `blocked_topics`.
- **pending_admin_actions** — ожидающие действия админа: тип (HANDOFF / CANCEL_APPROVAL / OTHER), `conversation_id`, `client_phone`, `case_id`, `approval_id`, `reminder_count`, `status` (OPEN | DONE).
- **agent_global_state** — глобальные флаги (ключ `enabled`).
- **telegram_admins** — допуск к командам бота: `telegram_user_id` (PK), `display_name`, `is_enabled`.

**Расширены таблицы из 002:**

- **handoff_cases** — добавлены `status`, `resolved_at`, `admin_response`.
- **conversation_messages** — добавлен индекс для дедупа по `(conversation_id, text_hash, ts)`.

**Семена в admin_config** (миграция 004): часы работы, дебаунс WhatsApp, takeover, напоминания, пороги агента. При отсутствии ключа в БД используются значения из `.env`.

### 2.2. Сервис Orchestrator (Node.js, TypeScript, Fastify)

**Структура кода:**

- **server.ts** — точка входа: пул БД, `initConfig`, Fastify, вебхук WhatsApp, дебаунс-процессор, запуск Telegram-бота и воркера напоминаний.
- **config.ts** — конфиг: сначала БД (`admin_config`), при отсутствии — значения из `initConfig` (ENV). Кэш с TTL.
- **db.ts** — подключение к Postgres (PGHOST, PGPORT, PGDATABASE, PGUSER, PGPASSWORD).

**Маршруты:**

- `GET /health` — проверка живости.
- `POST /webhooks/whatsapp` — приём вебхуков Meta (см. ниже).

**Сервисы:**

- **conversation** — создание/обновление диалога, смена состояния (BOT_ACTIVE, BOT_PAUSED, ADMIN_TAKEOVER, AWAITING_ADMIN, IGNORED), приоритет состояний, `getConversationByPhone` для команд бота.
- **messageStore** — сохранение сообщений с дедупом по `message_id` или по (ts, direction, text_hash); выборка последних N сообщений.
- **ignoreList** — проверка/добавление/снятие игнора по номеру (IGNORE / ADMIN_ONLY).
- **behaviorOverrides** — чтение и изменение правил по номеру (language_preference, tone_profile, force_handoff, notes_for_agent).
- **handoff** — создание кейса в `handoff_cases`, добавление записи в `pending_admin_actions`, получение открытых действий, отметка DONE по approval_id, обновление счётчика напоминаний, получение кейса по id.
- **debounce** — очередь по `conversation_id`: сбор сообщений в окно (debounce_ms, quiet_ms, max_debounce_total_ms, max_buffered_messages), затем один вызов процессора батча.
- **agentProcessor** — обработка батча: проверка состояния (только BOT_ACTIVE отвечает), force_handoff и ключевые слова (cancel, payment, complaint и т.д.), часы работы, вызов MCP (например `admin.get_upcoming_appointments_by_phone`), при необходимости создание handoff + pause-сообщение клиенту.
- **mcpClient** — HTTP-вызовы Gateway: `callMcp`, `createHandoffViaMcp`, `approveViaMcp`, `rejectApproval`.
- **whatsappSend** — отправка текста в WhatsApp через Meta Graph API (WHATSAPP_ACCESS_TOKEN, WHATSAPP_PHONE_ID).

**Утилиты:**

- **e164** — нормализация и проверка номера в E.164.
- **businessHours** — проверка «в рабочих часах» (timezone, start, end) и текст ночного сообщения.

### 2.3. Telegram-бот (Telegraf)

- Запускается внутри процесса Orchestrator при наличии `TELEGRAM_BOT_TOKEN`.
- **Проверка прав:** каждый запрос проверяет, что `ctx.from.id` есть в `telegram_admins` и `is_enabled = true`; иначе ответ «Not authorized.»

**Команды (в личке с ботом, E.164 для телефона):**

| Команда | Действие |
|--------|----------|
| `/takeover +<phone>` | Ставит состояние ADMIN_TAKEOVER, бот перестаёт отвечать. |
| `/resume +<phone>` | BOT_ACTIVE — бот снова отвечает. |
| `/pause +<phone>` | BOT_PAUSED — бот не отвечает. |
| `/ignore +<phone> [IGNORE\|ADMIN_ONLY] [reason]` | Добавляет номер в игнор (только сохранение или сохранение + пинг в группу логов). |
| `/unignore +<phone>` | Убирает номер из игнора. |
| `/status +<phone>` | Состояние диалога, последние 10 сообщений, открытые pending actions. |
| `/case <case_id>` | Информация по кейсу хендофа. |
| `/rules +<phone>` | Текущие переопределения по номеру. |
| `/setrule +<phone> key=value` | Установка language_preference, tone_profile, force_handoff, notes_for_agent. |
| `/approve <approval_id>` | Вызов MCP approve и пометка pending action как DONE. |
| `/reject <approval_id>` | Reject в MCP и DONE. |

**Куда шлёт бот:**

- **TG_LOGS_GROUP_CHAT_ID** (или `admin_config`: `telegram.logs_group_chat_id`) — ошибки, пинги ADMIN_ONLY и т.п.
- **TG_SUMMARY_GROUP_CHAT_ID** (или `telegram.summary_group_chat_id`) — саммари хендофов и напоминания воркера.

### 2.4. Воркер напоминаний

- Запускается при старте Orchestrator (`void startReminderWorker(...)`).
- Первый запуск — через `admin_reminder.first_after_minutes` (по умолчанию 10), дальше — каждые `admin_reminder.repeat_every_minutes` (15).
- Для каждой OPEN-записи в `pending_admin_actions` с `reminder_count < admin_reminder.max_reminders` проверяет: прошло ли достаточно времени с создания или с последнего напоминания; если да — шлёт текст в группу саммари и увеличивает `reminder_count` / обновляет `last_reminded_at`.

### 2.5. Конфигурация

**Источники (приоритет):** значения из таблицы **admin_config** перекрывают значения, переданные в **initConfig** (из `.env`).

**Ключи из миграции 004 (admin_config):**  
business_hours.timezone, start, end, night_message_enabled; whatsapp.debounce_ms, quiet_ms, max_buffered_messages, max_debounce_total_ms; takeover.auto_enabled, auto_ttl_minutes, auto_sender_detection_mode; admin_reminder.enabled, first_after_minutes, repeat_every_minutes, max_reminders; agent.confidence_threshold, max_clarifying_questions.

**Из .env в initConfig:**  
MCP_GATEWAY_URL, TELEGRAM_BOT_TOKEN, DEFAULT_COMPANY_ID, WHATSAPP_ACCESS_TOKEN, WHATSAPP_PHONE_ID, WEBHOOK_VERIFY_TOKEN, telegram.logs_group_chat_id (= TG_LOGS_GROUP_CHAT_ID), telegram.summary_group_chat_id (= TG_SUMMARY_GROUP_CHAT_ID), ADMIN_APPROVE_KEY.

## 3. Как это работает по шагам

### 3.1. Входящее сообщение WhatsApp

1. Meta шлёт **POST /webhooks/whatsapp** с телом от WhatsApp Cloud API.
2. Оркестратор парсит `entry[].changes[].value.messages[]`, приводит к полям: conversation_id (wa_<phone_number_id>_<from>), client_phone (E.164), ts, text, message_id.
3. Номер проверяется на E.164; сообщение сохраняется в **conversation_messages** (дедуп по message_id или ts+direction+text_hash), обновляется **conversations** (last_inbound_at), при необходимости создаётся диалог в BOT_ACTIVE.
4. Проверка **agent_ignore_phones**: если IGNORE — только сохранение, в дебаунс не попадает. Если ADMIN_ONLY — сохранение + при включённой настройке пинг в TG_LOGS_GROUP, в дебаунс не попадает.
5. Иначе **conversation_id** и данные сообщения кладутся в **дебаунс-очередь**.

### 3.2. Дебаунс

- Сообщения по одному conversation_id копятся в памяти.
- Таймер: по умолчанию окно ~20 с (debounce_ms), продлевается при новых сообщениях до max_debounce_total_ms; при тишине quiet_ms или при достижении max_buffered_messages батч «закрывается».
- Для батча вызывается **agentProcessor.processBatch**.

### 3.3. Обработка батча (agentProcessor)

1. Загружается диалог из **conversations**. Если состояние не **BOT_ACTIVE** — бот не отвечает, выход.
2. Проверяется **client_behavior_overrides**: при `force_handoff = true` — создаётся handoff, AWAITING_ADMIN, клиенту уходит pause-сообщение, в TG_SUMMARY — саммари.
3. Проверка ключевых слов (cancel, payment, complaint и т.д.): при совпадении — то же: handoff + AWAITING_ADMIN + сообщение клиенту.
4. Проверка **рабочих часов** (timezone, start, end из конфига): вне часов — клиенту отправляется ночное сообщение («We are available 08:00–20:00…»), состояние AWAITING_ADMIN, саммари в TG_SUMMARY.
5. В рабочих часах без хендофа — вызов **MCP** (например получение записей по телефону), формирование ответа и отправка в WhatsApp; при ошибке MCP — создание handoff и AWAITING_ADMIN.

Handoff: запись в **handoff_cases**, запись в **pending_admin_actions** (type HANDOFF), смена состояния на AWAITING_ADMIN, отправка клиенту паузового сообщения, по возможности вызов MCP handoff.create_case, саммари в TG_SUMMARY.

### 3.4. Верификация вебхука Meta

- При подписке Meta шлёт **GET** с `hub.mode=subscribe`, `hub.verify_token`, `hub.challenge`.
- Оркестратор сравнивает `hub.verify_token` с тем, что возвращает `getWebhookVerifyToken()` (из конфига, по сути WEBHOOK_VERIFY_TOKEN). При совпадении в ответе отдаётся `hub.challenge`.

### 3.5. Telegram: кто может пользоваться командами

- В **telegram_admins** должны быть добавлены `telegram_user_id` (и при необходимости display_name), `is_enabled = true`.
- Команды принимаются только от этих пользователей; иначе бот отвечает «Not authorized.»

## 4. Что нужно настроить у вас

### 4.1. Файл `.env`

- **Postgres:** при локальном запуске оркестратора — PGHOST, PGPORT, PGDATABASE, PGUSER, PGPASSWORD (в Docker они заданы в docker-compose).
- **Orchestrator:**  
  ORCHESTRATOR_PORT, MCP_GATEWAY_URL,  
  TELEGRAM_BOT_TOKEN, TG_LOGS_GROUP_CHAT_ID, TG_SUMMARY_GROUP_CHAT_ID,  
  WHATSAPP_ACCESS_TOKEN, WHATSAPP_PHONE_ID, WEBHOOK_VERIFY_TOKEN,  
  DEFAULT_COMPANY_ID, ADMIN_APPROVE_KEY.

Подробнее про получение токена бота и ID групп — в runbook и в комментариях `.env.example`.

### 4.2. База данных

- Добавить админов в **telegram_admins** (хотя бы одного), иначе все команды бота будут «Not authorized.»

### 4.3. Meta (WhatsApp)

- В приложении Meta настроить вебхук: URL на ваш публичный адрес `.../webhooks/whatsapp`, verify token = тот же, что в WEBHOOK_VERIFY_TOKEN.
- Подписать события **messages**.
- Взять WHATSAPP_ACCESS_TOKEN и WHATSAPP_PHONE_ID из раздела WhatsApp приложения.

## 6. Краткая схема потока

```
WhatsApp (Meta) → POST /webhooks/whatsapp
       → нормализация, сохранение сообщения, проверка игнора
       → дебаунс-очередь
       → по таймеру: processBatch
            → состояние BOT_ACTIVE? → часы? → MCP или handoff
            → ответ в WhatsApp (whatsappSend) и/или запись в handoff_cases + pending_admin_actions
            → уведомления в TG_SUMMARY / TG_LOGS

Telegram: команды от пользователей из telegram_admins
       → изменение state (takeover/resume/pause), игнор, правила, approve/reject
       → ответы в личку; логи/саммари — в группы по chat_id из конфига

Reminder worker: раз в N минут проверяет pending_admin_actions (OPEN)
       → шлёт напоминания в TG_SUMMARY и обновляет reminder_count
```

Всё описанное уже реализовано в коде и настроено через БД и `.env` как выше.

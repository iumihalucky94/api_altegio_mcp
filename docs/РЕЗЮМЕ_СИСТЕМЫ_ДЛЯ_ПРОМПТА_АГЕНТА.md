# Полное описание системы: связки, данные, последовательности (для промпта главного ИИ-агента)

Документ описывает всё, что сделано: кто что хранит, от кого что ждёт, как компоненты связаны и в какой последовательности всё работает. Цель — использовать это при проектировании главного ИИ-агента и его промпта.

---

## 1. Архитектура: три слоя

```
[ Клиент WhatsApp ]  ←→  [ MCP Gateway: WhatsApp Web ]  ←→  [ Orchestrator ]
         ↑                           ↑                              ↑
    телефон                    контейнер gateway              контейнер orchestrator
                               (транспорт + CRM)              (мозг: FSM, логика, Telegram)
                                                                     ↑
                                                              [ Postgres: одна БД ]
```

- **Postgres** — одна база для Gateway и Orchestrator (миграции 001–004).
- **Gateway (MCP)** — только транспорт и CRM: WhatsApp Web (whatsapp-web.js), приём/отправка сообщений, вызовы Altegio (MCP tools), апрувы. Бизнес-логику не содержит.
- **Orchestrator** — «мозг»: приём событий от Gateway, FSM диалогов, дебаунс, решение «отвечать / handoff / молчать», вызов MCP для CRM, Telegram-админка, напоминания. Сюда же позже подключается главный ИИ-агент (генерация ответов, решение о handoff и т.д.).

## 2. Контейнеры и порты

| Сервис        | Порт  | Роль |
|---------------|-------|------|
| postgres      | 5434→5432 | БД: все таблицы (mcp_requests, admin_config, conversations, conversation_messages, handoff_cases, pending_admin_actions, agent_ignore_phones, client_behavior_overrides, telegram_admins и др.). |
| redis         | 6380→6379 | Опционально (rate limit и кэши в Gateway). |
| gateway       | 3030  | WhatsApp Web-сессия, POST /whatsapp/send, POST /whatsapp/qr, POST /mcp (CRM), POST /approvals/..., пересылка входящих в Orchestrator. |
| orchestrator  | 3031  | POST /ingest/whatsapp-web, GET /health, логика диалогов, Telegram-бот, воркер напоминаний. |

## 3. Поток данных: входящее сообщение WhatsApp

Последовательность от «клиент написал в WhatsApp» до «ответ или handoff».

1. **Клиент** пишет в WhatsApp на номер, привязанный к WhatsApp Web в Gateway.
2. **Gateway (WhatsApp Web)** получает сообщение в браузерной сессии (whatsapp-web.js), парсит: от кого (номер), текст, timestamp, message_id.
3. **Gateway** шлёт в Orchestrator:
   - **Куда:** `POST {ORCHESTRATOR_INGEST_URL}/ingest/whatsapp-web`
   - **Тело:** `provider: "whatsapp-web", provider_message_id, client_phone_e164, text, ts_iso, raw_json`
   - **Заголовок:** `x-internal-token: WA_WEB_INTERNAL_TOKEN` (то же значение, что MCP_INTERNAL_TOKEN в Orchestrator).
4. **Orchestrator** (ingest):
   - Проверяет токен (если задан).
   - Нормализует номер в E.164.
   - Определяет `conversation_id` = `wa_web_{digits}` (один диалог на номер).
   - Пишет в **conversation_messages**: direction=in, author=client, text, ts, message_id (дедуп по message_id или ts+direction+hash).
   - Создаёт/обновляет **conversations**: last_inbound_at, при отсутствии — state=BOT_ACTIVE.
   - Проверяет **agent_ignore_phones**: если IGNORE — только сохранение, дальше ничего. Если ADMIN_ONLY — сохранение + пинг в TG_LOGS_GROUP, ответа нет.
   - Иначе кладёт событие в **дебаунс-очередь** по conversation_id.
5. **Дебаунс** (в Orchestrator):
   - Копит входящие по conversation_id в окно (debounce_ms, quiet_ms, max_debounce_total_ms, max_buffered_messages из admin_config).
   - По истечении окна вызывает **обработчик батча** (agentProcessor) один раз на накопленные сообщения.
6. **Обработка батча** (Orchestrator, место для подключения ИИ):
   - Загружает **conversations** по conversation_id.
   - Если state ≠ BOT_ACTIVE — бот не отвечает (выход).
   - Проверяет **client_behavior_overrides**: force_handoff → сразу handoff (см. ниже).
   - Проверка ключевых слов (cancel, payment, complaint и т.д.) → handoff.
   - Проверка **рабочих часов** (admin_config: business_hours.timezone, start, end). Вне часов — отправка ночного сообщения клиенту, state=AWAITING_ADMIN, саммари в Telegram.
   - В рабочих часах: вызов **MCP** (например `admin.get_upcoming_appointments_by_phone`), формирование ответа (сейчас захардкоженные фразы), отправка в WhatsApp через Gateway и **запись исходящего в conversation_messages** (direction=out, author=agent), обновление last_outbound_at в conversations.
   - При ошибке MCP или при handoff — создание кейса, паузовое сообщение клиенту, запись исходящего в БД.

Итого: входящие всегда пишутся в БД на шаге 4; исходящие от бота — при каждой успешной отправке (шаг 6), в ту же таблицу conversation_messages.

## 4. Поток данных: исходящее сообщение в WhatsApp

Кто хочет отправить текст клиенту — идёт только через Orchestrator → Gateway.

1. **Orchestrator** (agentProcessor или другой сервис) вызывает `sendWhatsAppMessage(toPhone, text, conversationId, logger)`.
2. Внутри: **HTTP POST** на Gateway `{MCP_GATEWAY_URL}/whatsapp/send`, тело `to_phone_e164, text, conversation_id`, заголовок `x-internal-token: MCP_INTERNAL_TOKEN`.
3. **Gateway** проверяет токен, отправляет сообщение через WhatsApp Web (whatsapp-web.js), возвращает `{ ok, provider_message_id }`.
4. **Orchestrator** при ok пишет в **conversation_messages** (direction=out, author=agent, message_id=provider_message_id) и обновляет **conversations.last_outbound_at**.

Исходящие, отправленные не через Orchestrator (например ручной вызов curl на /whatsapp/send), в БД оркестратора не попадают — резервная копия переписок формируется только из трафика, проходящего через оркестратор.

## 5. База данных: что где лежит

Одна БД Postgres, общая для Gateway и Orchestrator.

### 5.1. Диалоги и сообщения (резервная копия переписок)

- **conversations**
  - Один диалог на пару (канал + клиент). Для WhatsApp Web: conversation_id = `wa_web_{phone_digits}`, client_phone = E.164.
  - state: BOT_ACTIVE | BOT_PAUSED | ADMIN_TAKEOVER | AWAITING_ADMIN | IGNORED.
  - last_inbound_at, last_outbound_at, state_updated_at, language_hint, takeover_until, metadata_json.

- **conversation_messages**
  - Резервная копия всех сообщений, которые оркестратор принял (in) или отправил (out).
  - Поля: conversation_id, client_phone, ts, direction (in/out), author (client/agent/admin), text, message_id (от провайдера), text_hash, locale, metadata.
  - Дедуп: по message_id или по (conversation_id, ts, direction, text_hash).

### 5.2. Правила и ограничения

- **agent_ignore_phones** — по номеру: IGNORE (только писать в БД, не отвечать) или ADMIN_ONLY (писать в БД + уведомить админа в Telegram).
- **client_behavior_overrides** — по номеру: language_preference, tone_profile, force_handoff, notes_for_agent, blocked_topics. Используются для решений бота и (в будущем) для промпта ИИ.
- **telegram_admins** — кто может вызывать команды бота (telegram_user_id, is_enabled).

### 5.3. Handoff и ожидающие действия

- **handoff_cases** — кейс передачи человеку: conversation_id, client_phone, summary, question_to_admin, status (OPEN/RESOLVED/CLOSED), created_at, resolved_at, admin_response.
- **pending_admin_actions** — что ждёт действий админа: type (HANDOFF, CANCEL_APPROVAL, OTHER), conversation_id, client_phone, case_id, approval_id, status (OPEN/DONE), reminder_count, last_reminded_at. Воркер напоминаний в Telegram опирается на эту таблицу.

### 5.4. Конфиг и глобальное состояние

- **admin_config** — ключ–значение (value_json). Приоритет: значение из БД перекрывает ENV. Ключи: business_hours.*, whatsapp.debounce_ms и др., agent.confidence_threshold, telegram.logs_group_chat_id, telegram.summary_group_chat_id и т.д.
- **agent_global_state** — ключ (например enabled), value_bool. Глобальное вкл/выкл бота.

### 5.5. Аудит и CRM (Gateway)

- **mcp_requests** — лог вызовов MCP (request_id, company_id, tool, decision, actor и т.д.).
- Апрувы и политики — свои таблицы/механизмы в Gateway (для CRM и апрувов).

## 6. Что от кого ждёт

- **Gateway ждёт от Orchestrator:**
  - Вызовы POST /whatsapp/send (тело + x-internal-token) для отправки сообщений.
  - Вызовы POST /mcp с envelope (tool, payload, company_id, request_id и т.д.) для CRM.
  - Вызовы approve/reject для апрувов (по существующему API).

- **Orchestrator ждёт от Gateway:**
  - POST /ingest/whatsapp-web с нормализованным телом и токеном — единственный канал входящих WhatsApp.
  - Ответы от /whatsapp/send (ok, provider_message_id) и от /mcp (decision, result).

- **Orchestrator ждёт от БД:**
  - Актуальные conversations, conversation_messages, agent_ignore_phones, client_behavior_overrides, handoff_cases, pending_admin_actions, admin_config, telegram_admins.

- **Telegram-бот ждёт:**
  - Сообщения только от пользователей, чей telegram_user_id есть в telegram_admins с is_enabled=true. Команды: /takeover, /resume, /pause, /ignore, /unignore, /status, /case, /rules, /setrule, /approve, /reject (E.164 для номера где нужно).

- **Воркер напоминаний ждёт:**
  - OPEN-записи в pending_admin_actions и конфиг admin_reminder.* (first_after_minutes, repeat_every_minutes, max_reminders). Шлёт напоминания в TG_SUMMARY_GROUP.

---

## 7. Состояния диалога (FSM) и приоритет

Состояния: **BOT_ACTIVE**, **BOT_PAUSED**, **ADMIN_TAKEOVER**, **AWAITING_ADMIN**, **IGNORED**.

Приоритет (кто «сильнее»): IGNORED > ADMIN_TAKEOVER > BOT_PAUSED > AWAITING_ADMIN > BOT_ACTIVE.

- Ответ бота разрешён только при **state = BOT_ACTIVE**.
- Переключение состояний: через Telegram-команды (/takeover, /resume, /pause) или из логики (handoff → AWAITING_ADMIN; при необходимости можно выставлять BOT_PAUSED и т.д.).

---

## 8. Handoff: последовательность

1. В agentProcessor решено «нужен handoff» (ключевые слова, force_handoff, ошибка MCP и т.д.).
2. Создаётся запись в **handoff_cases** (summary, question_to_admin, status=OPEN).
3. Создаётся запись в **pending_admin_actions** (type=HANDOFF, case_id, status=OPEN).
4. **conversations.state** для этого conversation_id ставится в AWAITING_ADMIN.
5. Клиенту уходит паузовое сообщение (например «I forwarded your message to our team…») и оно пишется в conversation_messages как исходящее.
6. Опционально вызов MCP handoff.create_case (если нужна интеграция с CRM).
7. В Telegram (TG_SUMMARY_GROUP) уходит краткое саммари с case_id и контекстом.
8. Воркер напоминаний периодически напоминает админам о OPEN pending_admin_actions в Telegram.

Админ может посмотреть кейс (/case), ответить клиенту вручную (в WhatsApp или через другой канал), закрыть кейс и при необходимости вернуть бота (/resume).

---

## 9. Конфиг: откуда берутся значения

- **Источник:** сначала таблица **admin_config** (по ключу), при отсутствии — значения, переданные при init (из ENV). Кэш с TTL ~15 с.
- **Важные ключи для агента и промпта:** business_hours.*, agent.confidence_threshold, agent.max_clarifying_questions; для переопределений по клиенту — client_behavior_overrides в БД (language_preference, tone_profile, notes_for_agent, blocked_topics, force_handoff).

## 10. Где в коде «вставить» главного ИИ-агента

- **Точка входа:** обработка батча в **Orchestrator**, сервис **agentProcessor** (или выделенный слой «agent»).
- **Вход агента на один батч:**
  - conversation_id, client_phone;
  - последние N сообщений из **conversation_messages** (уже есть getLastMessages);
  - текущее состояние из **conversations** (state, last_inbound_at и т.д.);
  - правила из **client_behavior_overrides** (язык, тон, notes_for_agent, blocked_topics, force_handoff);
  - рабочие часы и флаг «сейчас внутри/вне часов»;
  - при необходимости — результат вызова MCP (например список записей по телефону).
- **Выход агента (что нужно принять решение):**
  - отвечать или нет;
  - если отвечать — текст ответа (и возможно выбор MCP-действия: запись, отмена и т.д.);
  - если не отвечать / передать человеку — флаг handoff и краткие summary + question_to_admin для handoff_cases.
- **После решения оркестратор:**
  - при ответе — вызов sendWhatsAppMessage (и уже реализованная запись в conversation_messages и last_outbound_at);
  - при handoff — текущая последовательность (handoff_cases, pending_admin_actions, state=AWAITING_ADMIN, паузовое сообщение, уведомления в Telegram).

То есть «мозг» (ИИ с промптом) должен получать структурированный контекст из БД и правил и возвращать структурированное решение (ответ / handoff + тексты), а всё остальное (FSM, запись в БД, вызов Gateway, Telegram) уже сделано и не меняется.

## 11. Краткая схема для промпта ИИ-агента

- **Роль:** ассистент по WhatsApp для бизнеса (запись/отмена, информация), в рабочих часах отвечает сам, вне — короткое ночное сообщение и передача человеку.
- **Вход:** история последних сообщений (in/out), номер клиента, правила по клиенту (язык, тон, заметки, запрещённые темы), факт «в рабочих часах / нет», при необходимости данные CRM (записи по телефону).
- **Ограничения:** не придумывать факты о записях; при сомнениях или запросе отмены/сложном запросе — handoff; соблюдать язык и тон из overrides; не затрагивать blocked_topics кроме нейтрального перенаправления.
- **Выход:** один из: (1) текст ответа клиенту (+ опционально вызов какого MCP tool), (2) handoff с кратким summary и question_to_admin.

# Оценка промпта (ChatGPT) для улучшения Altegio AI-агента

Документ: оценка присланного промпта, его смысл, сильные/слабые стороны и конкретные улучшения с учётом текущего кода.

---

## 1. В чём суть промпта

Промпт задаёт **архитектурную и продуктовую эволюцию** текущего MVP:

- **Не** «напиши код с нуля», а **управляемый рефакторинг** к production-ready агенту.
- **Три кита:** (1) язык ответа = язык клиента, без смешивания; (2) бизнес-правила и слоты — в коде/конфиге/политиках, не только в промпте; (3) заложить цикл review и развития агента.
- **Целевая модель:** один orchestrator + skills/modules + policies в БД + единый response writer + audit и review loop.
- **Новая модель данных:** расширение БД (conversations, messages, events, reviews, scenarios, scenario_policies, prompt_versions, corrections).
- **Поэтапность:** анализ → модель данных → рефакторинг оркестрации → основа review loop → decision guards → итог.

То есть промпт пытается превратить «работающий, но жёстко зашитый» бот в **управляемую агентную систему** с контролем языка, сценариев, автономии и возможностью улучшения по данным.

## 2. Оценка по критериям

### Сильные стороны

| Критерий | Оценка | Комментарий |
|----------|--------|-------------|
| Соответствие реальности проекта | Высокая | Упоминаются wa-service, orchestrator, gateway, MCP, Postgres, Redis; flow (ingest → conversation → policy → response) совпадает с тем, что есть. |
| Конкретика проблем | Высокая | Язык ответа, правила в промпте, слоты без проверки, автономия по сценариям, review — всё это есть в коде (см. ниже). |
| Целевая архитектура | Хорошая | Один оркестратор, skills, policies в БД, один response writer, audit — разумно и не «зоопарк агентов». |
| Модель данных | Продуманная | Разделение conversations / messages / events / reviews / scenario_policies даёт место для эволюции и аналитики. |
| Поэтапность | Хорошая | Анализ → схема БД → рефактор → review → guards → итог снижает риск сломать работающий flow. |
| Критические правила | Уместные | Language lock, no slot confirm without validation, policy checks, no random hardcoded phrases — прямо бьют по текущим слабостям. |

### Слабые стороны / риски

| Критерий | Оценка | Комментарий |
|----------|--------|-------------|
| Объём за один заход | Высокий | Один ответ модели может «размазаться»: поверхностный анализ, абстрактные таблицы, мало конкретных патчей. Лучше разбить на 2–3 промпта (анализ → план → этап 1). |
| Дублирование с существующей схемой | Средний | Уже есть `conversations`, `conversation_messages`, `handoff_cases`. Промпт вводит новые имена полей и таблиц (например `conversation_events`, `conversation_reviews`) без явного указания: мигрировать текущие таблицы или добавить новые и связать. Нужно явно сказать «расширить текущие conversations/conversation_messages, добавить только недостающие таблицы». |
| Skills vs текущий код | Средний | В коде уже есть аналоги: intent (classifyIntent), language (detectLanguage), handoff, booking (get_free_slots, validate_slot). Промпт не привязан к ним: «раздели на модули» может привести к переименованию без пользы. Лучше явно: «выдели навыки на базе существующих agentProcessor, intent, handoff, bookingContext». |
| Приоритеты | Размыты | Всё «важно». Для исполнителя (модель или человек) полезно: «Сначала язык и слоты (клиент видит сразу), потом scenario_policies, потом review». |
| Формат вывода | Общий | «Покажи diff/patch plan» — модель часто даёт описания, а не готовые патчи. Лучше: «Для Этапа 1 выдай: (1) список файлов и изменений по каждому, (2) один конкретный patch для файла X». |

### Соответствие текущему коду (кратко)

- **Язык:** в системном промпте уже есть «Always respond in the same language the client used»; при этом в `agentProcessor.ts` есть **жёстко заданные немецкие** фразы при ошибке бронирования и fallback (`Leider ist bei der Buchung...`, `Vielen Dank für Ihre Nachricht...`). Проблема реальная.
- **Слоты:** проверка через `get_free_slots` и `validate_slot` уже есть; промпт правильно требует «никогда не подтверждать слот без code/tool».
- **Правила в промпте:** часть логики (handoff triggers, confidence threshold) в коде; сценариев и политик по сценариям в БД пока нет — расширение уместно.
- **БД:** есть `conversations` (state, language_hint, …), `conversation_messages`, `handoff_cases`, `agent_ignore_phones`, `client_behavior_overrides`. Нет: conversation_events, reviews, scenario_policies, prompt_versions, corrections — предложенная схема дополняет, а не заменяет с нуля.

## 3. Рекомендуемые улучшения промпта

### 3.1. Явно привязать к репозиторию

Добавить в начало (после контекста про Docker/orchestrator/gateway):

- «В репозитории уже есть: таблицы `conversations`, `conversation_messages`, `handoff_cases`; сервисы `agentProcessor`, `intent` (classifyIntent, detectLanguage), `handoff`, `bookingContext`, `aiAgent`, `kb`. Сначала изучи их и опиши, что уже реализовано; новые сущности и модули предлагай как расширение, а не замену без необходимости.»

Так снижается риск «придумать архитектуру в вакууме» и сломать рабочий flow.

### 3.2. Разбить на этапы с одним фокусом на ответ

Вместо одного гигантского «сделай всё» лучше несколько промптов:

- **Промпт 1 (анализ):** «Изучи репозиторий. Выдай: (1) текущая архитектура в виде списка компонентов и потоков, (2) список конкретных мест с захардкоженным языком/фолбэками, (3) где уже есть проверка слотов, (4) слабые места и quick wins. Без реализации.»
- **Промпт 2 (план и язык):** «По результатам анализа: (1) план изменений по приоритетам, (2) конкретный план централизованного локализатора и список строк для замены в agentProcessor (и других файлах).»
- **Промпт 3 (этап реализации):** «Реализуй только Этап X: [конкретный список задач]. Дай патчи/изменения по файлам.»

Так модель даёт более конкретный и проверяемый результат.

### 3.3. Уточнить модель данных относительно текущей БД

Добавить явное требование:

- «Текущие таблицы: `conversations` (conversation_id, client_phone, state, last_inbound_at, last_outbound_at, language_hint, …), `conversation_messages` (conversation_id, direction, author, text, message_id, …). Предложи только миграции, которые расширяют эти таблицы (новые колонки) и добавляют новые таблицы (conversation_events, conversation_reviews, scenario_policies и т.д.) с явными foreign key на существующие. Не переименовывай существующие таблицы без необходимости.»

Это уменьшает риск «новой схемы с нуля» и двойного хранения одних и тех же данных.

### 3.4. Зафиксировать приоритеты

Добавить блок:

- «Приоритет внедрения: (1) язык и локализация — клиент сразу видит корректный язык; (2) проверка слотов и запрет подтверждения без tool — безопасность; (3) scenario_policies и autonomy — управление поведением; (4) conversation_events и audit — наблюдаемость; (5) review/reviews/corrections — развитие агента.»

Можно в том же порядке раздавать этапы исполнителю.

### 3.5. Конкретизировать формат вывода по этапам

Для этапа с реализацией добавить:

- «Для каждого изменённого файла выдай: путь к файлу, цель изменения, и полный патч (или последовательность точных правок). Для миграций — полный SQL файл. Не ограничивайся общим описанием.»

Так проще сразу применять изменения в репозитории.

### 3.6. Явно про «не ломать»

Повторить в конце:

- «Не менять контракты ingest (POST /ingest/whatsapp-web), формат ответа AI-агента (decision, reply_text, handoff, mcp_calls), и не удалять debounce/processBatch без замены. Изменения только добавляют или расширяют поведение.»

Это страхует от поломки интеграции wa-service ↔ orchestrator и текущего AI-контракта.

## 4. Итоговая оценка

| Аспект | Оценка (1–5) | Комментарий |
|--------|--------------|-------------|
| Полнота целей | 5 | Покрыты язык, правила, слоты, автономия, review. |
| Соответствие коду | 4 | Хорошо, но нужно явно ссылаться на существующие таблицы и сервисы. |
| Реализуемость за один проход | 3 | Слишком много за раз; разбиение на этапы повысит качество. |
| Чёткость приоритетов | 3 | Всё важное; явный порядок приоритетов поможет. |
| Риск сломать текущее | 4 | Есть напоминание «не ломай», но лучше явно зафиксировать контракты и потоки. |

**Общая оценка: 4/5.** Промпт очень полезен как техническое задание на эволюцию архитектуры и хорошо попадает в реальные проблемы проекта. Чтобы получить от него максимальную пользу и меньше «воды», стоит: (1) привязать к текущему коду и БД, (2) разбить на этапы с одним фокусом на ответ, (3) задать приоритеты и (4) требовать конкретные патчи/миграции, а не только описание.

## 5. Конкретные места в коде под первый этап (язык)

Чтобы сразу было с чего начать (даже без ChatGPT):

- **agentProcessor.ts** (примерно строки 299, 317, 336): заменить захардкоженные строки:
  - `'Leider ist bei der Buchung etwas schiefgelaufen...'`
  - `'Leider kann ich die Buchung nicht automatisch bestätigen...'`
  - `'Vielen Dank für Ihre Nachricht. Unser Team meldet sich in Kürze bei Ihnen.'`  
  на вызовы централизованного локализатора по `language_hint` / `detected_language` (например `getMessage('booking_failed', lang)`, `getMessage('handoff_ack', lang)`).

- **Системный промпт** (aiAgentSystemPrompt.ts): правило про язык уже есть; полезно добавить: «All system fallback messages (booking failed, handoff acknowledgement, etc.) must be in the same language as the client's last message; the code will pass you the reply_language.»

После этого можно использовать доработанный промпт для следующих этапов (модель данных, scenario_policies, review loop).
